# 定期イラスト生成運用設計
## ローカルフォルダ・GitHub・Codex・ChatGPTプロジェクトの役割分担

- 対象リポジトリ: `Hiryu-zu/illustration-prompt-library`
- 目的: プロンプト図鑑を正本として維持しながら、専用チャットで定期的にイラストを生成し、結果を再利用可能な制作資産として蓄積する。
- 基本原則: **考える場所・正本を置く場所・編集する場所・画像を生成する場所・記録する場所を分ける。**

---

## 1. 全体像

```text
ローカルPC（作業場）
  ├─ VS Code
  ├─ 図鑑リポジトリのクローン
  ├─ 画像の一次保管
  ├─ 生成キューの編集
  └─ Git操作・動作確認
          │
          │ git push / git pull
          ▼
GitHub（共有可能な正本）
  ├─ プロンプト図鑑
  ├─ キャラクターアセット
  ├─ 生成キュー
  ├─ 採用済み画像と記録
  └─ 運用設計書
          │
          ├─ Codex がリポジトリを読んで修正・検証
          └─ ChatGPT プロジェクトが図鑑運用の文脈を保持
                    │
                    ▼
専用チャット「定期イラスト生成室」
  ├─ ChatGPTタスクの実行先
  ├─ 指定IDの画像生成
  ├─ 実行ログの作成
  └─ 採用 / 保留 / 再試行の一次判定
```

### 正本の優先順位

```text
GitHubの図鑑データ
  > ローカル作業フォルダ
  > ChatGPTプロジェクトの指示・添付資料
  > 専用生成チャットの会話ログ
  > タスク本文
```

- **プロンプト本文・採用状態・キャラクター正本はGitHubを最優先**とする。
- ローカルは編集と検証のための作業コピーであり、GitHubへ反映する前の変更を含んでよい。
- 専用生成チャットは実行記録を残す場所であり、設定の正本にはしない。
- タスク本文は「いつ、何をするか」を起動するだけの短い指示に限定する。

---

## 2. どこで何をするか

| 場所 | 主な役割 | 置くもの | 置かないもの |
|---|---|---|---|
| ローカルPC | 編集・検証・画像整理・Git操作 | 作業中のMD、画像、スクリプト、VS Code設定 | 採用状態だけをローカル限定で更新した正本 |
| GitHub | 図鑑の正本・履歴・共有・公開 | prompts、cards、modules、assets、outputs、docs、queue | 一時ファイル、未整理の大量画像、秘密情報 |
| Codex | 実装・構造改善・検証・差分作成 | 修正指示、受入基準、対象パス | 採用判定の最終決定、無制限な自動書換え |
| ChatGPTプロジェクト | ルールの固定・関連チャットの整理 | プロジェクト指示、運用メモ、専用生成チャット | 図鑑の唯一の正本、全画像の永続保管 |
| 専用生成チャット | 指定IDの生成と短い一次レビュー | 今回の生成条件、結果、簡易ログ | アイデア出し全般、長期設定の大改造 |
| ChatGPTタスク | 定期起動・通知・実行依頼 | 実行頻度、対象キュー、枚数上限 | 図鑑全文、キャラ正本全文、複雑な判断基準 |

## 3. ローカルフォルダの推奨構成

Windows PCでは、GitHubリポジトリを次のようにクローンして運用する。

```text
D:\Creative\illustration-prompt-library\
├─ index.html
├─ 研究ライブラリ.html
├─ プロンプトビューアー.html
├─ prompts\
│  ├─ library\                 # 採用済み・再利用可
│  ├─ reviewed\                # 検証済み・調整候補
│  └─ draft\                   # 草案・未検証
├─ cards\                       # 表現原理・知識カード
├─ modules\                     # 組み立て部品
├─ library-assets\
│  ├─ characters\              # キャラクター正本
│  ├─ worlds\                  # 背景・世界観アセット
│  └─ references\              # 参照画像の説明・利用条件
├─ defaults\                    # 共通補正・ネガティブ・既定値
├─ catalog\                     # 索引、採用基準、検索用データ
├─ queue\                       # 定期生成キューと履歴
│  ├─ generation-queue.json
│  └─ generation-history.json
├─ outputs\
│  ├─ candidates\              # 未採用の候補画像
│  ├─ reviewed\                # レビュー済み画像
│  └─ adopted\                 # 正式採用された画像
├─ docs\
│  ├─ 運用設計\
│  ├─ 書庫設計\
│  └─ 作業手順\
├─ tools\                       # 補助スクリプト
└─ .gitignore
```

### ローカルで行う作業

1. VS Codeでプロンプト、カード、モジュール、キューを編集する。
2. 新しい画像はまず `outputs/candidates/` へ置く。
3. 画像を確認して、採用候補なら `outputs/reviewed/`、正式採用なら `outputs/adopted/` へ移す。
4. 画像の相対パスをプロンプトMDの `preview_image` または `adopted_image` へ記録する。
5. `git status` で変更点を確認する。
6. 意味がまとまった単位でコミットし、GitHubへpushする。

### ローカルに置かないもの

- APIキー、ログイン情報、アクセストークン
- 使い捨ての大量ダウンロード
- 参照元が不明な画像
- 生成条件が分からない画像
- 正式採用前の画像を `adopted/` に入れること

## 4. GitHub図鑑の担当範囲

GitHubは**プロンプト図鑑の唯一の正本**とする。

### GitHubに保存するもの

- 完成プロンプト（採用済み・検証中・草案）
- 知識カードとモジュール
- キャラクター正本アセット
- 共通ネガティブ、補正、レビュー基準
- 生成キュー
- 生成履歴
- 採用画像とその紐付け情報
- 運用ルール、作業手順、設計書

### GitHubに反映するタイミング

| 変更 | pushの目安 |
|---|---|
| 新しい完成プロンプトを追加 | テスト前でも追加時にpush可 |
| 生成テストの所見を追加 | 1件の検証が終わった時点 |
| 採用画像を追加 | 採用判定後すぐ |
| キューを変更 | 次の実行対象を決めた時点 |
| HTML/CSS/JSを修正 | ローカルで画面確認後 |
| 大きい構造変更 | ブランチまたは分けたコミットで実施 |

### コミットメッセージ例

```text
prompt: add martial counter draft
review: adopt speed melee v2
asset: add xiaoling reference profile
queue: schedule next three prompt tests
docs: update generation workflow
ui: add status filter to research library
```

## 5. Codexの担当範囲

Codexは、図鑑を読んで**実装・検証・整理を行う技術担当**として使う。

### Codexに任せること

- 図鑑HTMLの検索・フィルタ・画面導線の改善
- JSON・front matter・索引の整合性チェック
- 生成キューのスキーマ作成
- 画像ファイル名・パスの検査
- `preview_image` / `adopted_image` のリンク切れ確認
- Markdownテンプレートの追加
- 生成履歴を図鑑用データへ整形
- Gitの差分確認、コミット候補の作成
- GitHub Pages上で崩れないための静的サイト検査
- 小規模なローカル補助ツールの作成

### Codexに任せないこと

- 画像の最終採用判断
- キャラクターidentityの最終確認
- プロンプトの目的を勝手に変更すること
- 既存の採用済みプロンプトを無断で上書きすること
- 一度に大量のファイルを削除・移動すること
- APIキーや秘密情報をファイルへ書き出すこと

### Codexへの依頼テンプレート

```text
リポジトリ内の `queue/generation-queue.json` と
`catalog/ライブラリ索引.json` を確認してください。

目的:
- 定期生成対象として不整合のあるIDを見つける。
- `eligible: true` だが画像比率、必要アセット、statusが不足している項目を列挙する。

制約:
- ファイルを変更しない。
- 結果は「問題 / 対象パス / 修正案」の表で示す。
- 採用・不採用の判断は行わない。
```

```text
`研究ライブラリ.html` に、完成プロンプト一覧の
「採用済みのみ」フィルタを追加してください。

受入基準:
- library / reviewed / draft を選択できる。
- 初期表示は全件。
- モバイル画面で横スクロールを発生させない。
- 既存の検索・タブ・hash遷移を壊さない。
- 変更前後の差分と確認手順を報告する。
```

### Codex運用ルール

1. 依頼時に必ず対象パスと受入基準を書く。
2. まず「読む・確認する」依頼を行い、次に「変更する」依頼を行う。
3. 大きな変更はブランチまたは小さなコミット単位に分ける。
4. 変更後は必ず差分、影響範囲、手動確認手順を出させる。
5. 採用済みデータは、ユーザーの明示指示なしに改変させない。

## 6. ChatGPTプロジェクトの担当範囲

ChatGPTプロジェクトは、図鑑運用のための**会話の作業場**である。

### プロジェクトに入れるもの

- プロジェクト指示
- 専用生成チャット
- 研究・設計チャット
- キャラ正本の概要
- 生成時の絶対ルール
- ファイルの置き場所の説明
- GitHubリポジトリURL
- よく使う依頼テンプレート

### プロジェクトに入れないもの

- GitHubを更新しないままの唯一の完成プロンプト
- 画像の正式保管場所
- 生成キューの唯一の最新版
- 個人的な秘密情報

### プロジェクト内のチャット分割

```text
プロンプト図鑑・定期生成
├─ 定期イラスト生成室｜図鑑実行チャット
│   └ 指定IDを生成し、短いレビューを残す
├─ 図鑑研究室｜カード・モジュール設計
│   └ 表現原理、知識カード、再利用部品を作る
├─ 図鑑編集室｜プロンプトと索引の整理
│   └ MD構造、タグ、採用基準、キューを整える
└─ Codex依頼室｜実装・修正依頼の下書き
    └ Codexへ渡す要件と受入基準を作る
```

### 普段の創作チャットとの境界

- 小鈴、リーファン、牙焔、ラヴィなどの物語・演出・構図の壁打ちは、普段の創作チャットで行う。
- そこから「図鑑に登録する価値がある」と判断した内容だけを、図鑑研究室へ移す。
- 定期イラスト生成室では新規アイデアを長く考えない。既存IDを実行する。

## 7. 専用チャット「定期イラスト生成室」のルール

### 目的

図鑑に登録されたプロンプトを、定期的かつ少量ずつ生成・検証する。

### 一回の実行単位

```text
1実行 = 原則1プロンプトID + 画像1枚 + 短いレビュー1件
```

### 実行前に確認する項目

- 図鑑ID
- 目的表現
- プロンプトの状態（library / reviewed / draft）
- 画像比率
- キャラクターアセットの要否
- 参考画像の要否
- 使用する補正
- 再試行回数

### 実行後に残すログ

```markdown
## 実行ログ

- 実行日時:
- 図鑑ID:
- プロンプト状態:
- 出力比率:
- 使用したキャラクターアセット:
- 参考画像:
- 使用した補正:
- 成功した点:
- 改善点:
- 判定: 採用 / 保留 / 再試行
- 次回の修正: 1〜3点
- 保存先:
```

### 禁止事項

- 同じ対象の無制限な連続生成
- 参照画像必須のキャラクターを、参照なしで自動生成すること
- 生成結果だけで正式採用扱いにすること
- 一回の失敗でプロンプトの目的を大きく変えること
- 画像に文字、ロゴ、透かし、可読UIを勝手に追加すること

## 8. ChatGPTタスクの担当範囲

ChatGPTタスクは**定期起動のきっかけ**として使う。

### タスクに書くこと

- 実行頻度
- 実行対象のキューまたは固定ID
- 一回あたりの枚数上限
- 実行後のログ形式
- 参照アセットが不足している場合の停止ルール

### タスクに書かないこと

- 図鑑全体の本文
- キャラクター正本全文
- 長いレビュー基準
- 生成画像の保存先を判断する複雑なロジック
- 自動で採用・GitHub更新を行う指示

### 初期タスク例

```text
毎週 火曜・金曜の21:00に、
「定期イラスト生成室｜図鑑実行チャット」で
生成キューの先頭にある対象を1件だけ実行する。

実行前に、図鑑ID、目的、状態、必要アセットを短く表示する。
参照画像またはキャラクター正本が必須で不足している場合は画像生成をせず、
「参照アセット待ち」として実行ログだけ残す。

生成後は、採用 / 保留 / 再試行を判定し、
改善点を最大3点だけ記録する。
```

### 初期運用の上限

- 頻度: 週2回
- 一回の出力: 1枚
- 同一IDの再試行: 最大3回
- キューの長さ: 3〜5件
- 正式採用: 人間の確認後のみ

## 9. 生成キューの設計

`queue/generation-queue.json` を、定期生成の対象を決める正本として使う。

```json
{
  "version": 1,
  "updated": "2026-06-27",
  "policy": {
    "max_images_per_run": 1,
    "retry_limit": 3,
    "allowed_prompt_status": ["library", "reviewed"],
    "require_reference_for_character_prompts": true,
    "human_approval_required_for_adoption": true
  },
  "queue": [
    {
      "job_id": "JOB-20260701-001",
      "prompt_id": "PRM-ACTION-SPEED-MELEE-001",
      "priority": 1,
      "status": "ready",
      "aspect_ratio": "16:9",
      "character_asset_id": null,
      "world_asset_id": null,
      "reference_required": false,
      "attempts": 0,
      "notes": "素手戦として可読性を再確認する"
    }
  ]
}
```

### キューの状態

| status | 意味 |
|---|---|
| `ready` | 次回生成してよい |
| `blocked` | アセット・設定不足で止める |
| `running` | 実行中または実行記録待ち |
| `review` | 生成済み、採否確認待ち |
| `done` | 採用または完了 |
| `archived` | 保留・終了、今は実行しない |

## 10. 画像とログの保存ルール

### ファイル名

```text
{prompt_id}__{YYYYMMDD}__v{n}__{state}.png

例:
PRM-ACTION-SPEED-MELEE-001__20260701__v1__candidate.png
PRM-ACTION-SPEED-MELEE-001__20260703__v2__adopted.png
```

### 保存先

| 状態 | 保存先 |
|---|---|
| 生成直後 | `outputs/candidates/` |
| レビュー対象 | `outputs/reviewed/` |
| 正式採用 | `outputs/adopted/` |
| 破棄しない失敗例 | `outputs/reviewed/rejected/` |

### 画像とプロンプトの紐付け

採用画像が出たら、対応するプロンプトMDへ最低限これを記録する。

```yaml
preview_image: outputs/adopted/PRM-ACTION-SPEED-MELEE-001__20260703__v2__adopted.png
adopted_image: outputs/adopted/PRM-ACTION-SPEED-MELEE-001__20260703__v2__adopted.png
test_status: reviewed
review_score: 86
review_status: adopted
generation_attempts: 2
updated: 2026-07-03
```

## 11. 日常運用フロー

### A. 新しい表現を思いついたとき

```text
普段の創作チャット
  → 図鑑研究室で原理・部品を整理
  → ローカルでカード / モジュール / 草案プロンプトを作成
  → GitHubへpush
  → 必要なら生成キューへ追加
```

### B. 定期生成する日

```text
ChatGPTタスク
  → 定期イラスト生成室
  → 指定IDを1枚生成
  → 一次レビューを残す
  → ローカルへ画像を保存
  → GitHub上の画像・MD・履歴を更新
  → Codexで整合性を確認
  → 人間が採用を最終決定
```

### C. 図鑑サイトを改善したいとき

```text
改善案を図鑑編集室で整理
  → Codex依頼文を作る
  → Codexに対象パス・受入基準つきで依頼
  → ローカルで画面確認
  → Git差分確認
  → GitHubへpush
```

## 12. 週次レビュー

毎週一回、15〜30分で次を確認する。

- [ ] キューが3〜5件に保たれている
- [ ] `blocked` の理由が明確である
- [ ] 生成画像に対応するログがある
- [ ] 採用画像に対応するプロンプトMDが更新されている
- [ ] `preview_image` のリンク切れがない
- [ ] 草案が増えすぎていない
- [ ] 同じ失敗を繰り返していない
- [ ] 次週に検証したい仮説が一文で書ける

## 13. 最初に整える順番

### 第1段階: 手動で回る形を作る

1. この運用設計を読む。
2. ローカルにリポジトリをクローンする。
3. `queue/` と `outputs/` の運用を始める。
4. 専用生成チャットを一つ作る。
5. 週2回・1回1枚のタスクを設定する。

### 第2段階: 図鑑を実行しやすくする

1. プロンプトに `automation` 情報を追加する。
2. 採用済み / 検証中 / 草案のフィルタを画面へ追加する。
3. キャラクター正本アセットを整える。
4. 生成ログテンプレートを統一する。

### 第3段階: Codexで半自動化する

1. JSON整合性チェックを作る。
2. 画像パスのリンク切れ検査を作る。
3. キューから候補一覧を作る小ツールを作る。
4. GitHub Pages上での表示テストを自動化する。

## 14. 最重要ルールまとめ

1. **GitHub図鑑が正本。**
2. **ローカルは編集・画像整理・検証の場。**
3. **Codexは技術作業を担当し、創作の最終判断はしない。**
4. **ChatGPTプロジェクトは会話とルールを整理する場。**
5. **定期生成は専用チャット一つに限定する。**
6. **タスクは起動役であり、図鑑全文を持たせない。**
7. **1実行1枚、改善は最大3点、同一IDの再試行は最大3回。**
8. **採用は必ず人間が最終確認する。**
9. **生成結果・画像・プロンプト・レビューを必ず紐付ける。**
10. **無関係な会話を定期生成室へ混ぜない。**
