# 現状と次の着手

## いま出来ているところ

### 1. 研究資料から書庫化する基本構造

既に `inbox` の研究資料から、知識カード、プロンプト部品、完成プロンプトへ展開する構造がある。

主な構成:

- `inbox/`
- `cards/`
- `modules/`
- `prompts/draft/`
- `catalog/`
- `outputs/`
- `templates/`

### 2. 研究ライブラリのビューワー

`研究ライブラリ.html` があり、GitHub Pages からスマホでも確認できる状態になっている。

現状の役割:

- プロンプト一覧を見る
- 検索する
- 詳細を開く
- プロンプト本文を確認する
- 作例画像があるものを表示する

### 3. アクション系プロンプトの階層化

アクション研究資料から、10ファミリー × 3段階のプロンプトが作られている。

ファミリー例:

- 高速戦・瞬間移動
- 重量級の一撃
- 武術・格闘技
- 剣戟
- 銃撃・射撃戦
- 魔法・必殺技
- 巨大怪獣・巨大ロボ
- スポーツ
- ダンス・舞踏
- レース・乗り物

### 4. 上位3件の生成テスト

上から3件のプロンプトについて、生成テスト画像が作成され、ビューワーで見られるようになっている。

対応済み:

- `PRM-ACTION-SPEED-MELEE-001`
- `PRM-ACTION-SPEED-TELEPORT-002`
- `PRM-ACTION-SPEED-TEMPLATE-003`

### 5. 作例表示の入口

プロンプトのメタ情報に `preview_image` を持たせ、ビューワーで作例を表示する仕組みがある。

これは今後のアルバム化の土台になる。

## まだ出来ていないところ

### 1. 実験場と書庫の分離

現在は `prompts/draft/` に多くのプロンプトがあり、未検証、検証中、書庫入りの違いがユーザー視点では分かりにくい。

必要な整理:

- テストプロンプト
- レビュー待ち
- 要改善
- 書庫入り
- アルバム化済み

を明確に分ける。

### 2. レビュー基準の明文化

作例を見て何をもって合格とするかが、まだ運用として固定されていない。

必要なレビュー観点:

- 目的表現が伝わるか
- 差し替えやすいか
- 専用キャラと組み合わせやすいか
- 破綻しやすい点が記録されているか
- デフォルト補正の要否が分かるか
- 再利用価値があるか

### 3. アセット管理

専用キャラ、衣装、背景、世界観、NG 指定などを独立アセットとして保存する仕組みはまだない。

必要なもの:

- アセット保存形式
- アセット一覧
- プロンプトとの組み合わせ指定
- 最終生成プロンプトへの反映

### 4. デフォルト補正のオンオフ

手、姿勢、顔、衣装などの崩れやすい部分を補正するデフォルト設定は、まだ分離管理されていない。

必要なもの:

- 補正カテゴリ
- プリセット
- positive / negative の分離
- UI 上のオンオフ
- プロンプトごとの推奨補正

### 5. SPA 的な操作体験

現在のビューワーはライブラリ閲覧が中心。

今後は、以下をタブやパネルで切り替える形にしたい。

```text
資料
テスト
書庫
アセット
生成
アルバム
```

### 6. アルバム保存

完成イラストと最終生成プロンプトをセットで保存する仕組みはまだない。

必要なもの:

- 完成画像
- 使用した書庫プロンプト
- 組み合わせたアセット
- 有効化した補正
- 最終生成プロンプト
- 所見
- 生成日

## 次にどこからやるか

次の着手は、いきなり大きな SPA 化ではなく、データ構造を先に固めるのがよい。

理由は、UI を先に作ると、あとで保存形式が変わった時に作り直しが増えるため。

## 推奨する次の順番

### Step 1. ステータス設計を入れる

まず、プロンプトを制作段階で分類できるようにする。

追加したいステータス:

```yaml
status: testing
status: needs_review
status: needs_revision
status: library
status: albumed
```

既存の `draft` との関係も整理する。

### Step 2. レビュー基準テンプレートを作る

プロンプトごとに、書庫入り判定ができるテンプレートを作る。

例:

```md
## レビュー

- 目的表現:
- 読みやすさ:
- 差し替えやすさ:
- 破綻リスク:
- 推奨デフォルト補正:
- 書庫入り判定:
```

### Step 3. デフォルト補正データを作る

`defaults/` を追加し、手、姿勢、顔、衣装、構図、NG 指定を JSON または Markdown で管理する。

最初は以下の5つから始める。

- 手・指
- 姿勢・骨格
- 顔・目
- 衣装破綻
- アクション向け補正

### Step 4. アセットの最小形式を作る

`assets/` または `library-assets/` を追加し、まず専用キャラを1件保存する形式を作る。

最初の対象:

- キャラ設定
- 衣装
- 背景
- NG 指定

### Step 5. ビューワーに「組み立て」欄を追加する

既存の `研究ライブラリ.html` に、最初は軽い機能として追加する。

追加したい UI:

- デフォルト補正のチェックボックス
- アセット選択
- 最終生成プロンプトのプレビュー
- コピー按钮

### Step 6. アルバム形式を作る

生成した完成イラストを保存するために、`albums/` を追加する。

まずは手動登録でよい。

保存単位:

```text
完成画像 + 最終生成プロンプト + 使用プロンプト + 使用アセット + 所見
```

## 最初の実装候補

最初に実装するなら、以下が最も効果が高い。

1. `defaults/` を作る
2. デフォルト補正を5カテゴリだけ登録する
3. 既存ビューワーの詳細画面にオンオフ欄を出す
4. 選択中プロンプト + 補正を結合してコピーできるようにする

これで、ユーザーはすぐに「書庫プロンプトを安全装置つきで使う」体験を得られる。

その後に、専用キャラアセットとアルバム保存へ進む。

## 判断

現時点では、書庫の中身を増やすより、先に「使うための構造」を整える段階。

次の優先順位は次の通り。

```text
1. ステータスとレビュー基準
2. デフォルト補正
3. アセット管理
4. 最終プロンプト組み立て UI
5. アルバム保存
```
